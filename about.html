游戏考古

B站频道以"消失游戏开发日志"系列为核心，专注追溯那些被取消、被雪藏、胎死腹中的游戏作品背后的开发故事。从《寂静岭P.T.》中小岛秀夫与科乐美的权力斗争，到《细胞分裂》如何被育碧一步步边缘化，再到《超越善恶2》创下游戏史上最长跳票纪录——每一部"消失的游戏"都承载着值得被记录的开发史。

目前该系列已发布6期，累计播放量超过13万，其中《细胞分裂》单期最高播放4.6万。最新一期探讨了国产沙盒游戏史上最抽象的欺诈事件，进一步拓展了系列的选题边界。

"游戏不只是代码和美术，它们是团队、资本、时机和运气的复杂博弈。每一部消失的游戏，都是一段被遗忘的历史。"

世界观构建

独立构建了「艾瑞斯大陆」奇幻世界观，累计设定文本约3.2万字。与常见的"剑与魔法"设定不同，这个世界观的核心张力来自三股力量的博弈：

  • 物理侧 — 以肉身与兵器为媒介的能力者，崇尚力量与荣誉
  • 魔法侧 — 以元素与秘法为媒介的能力者，追求知识与神秘
  • 研究侧 — 无能力的普通人，依靠制度、资本与知识居中制衡，实际掌控着大陆的政治与经济命脉

设定集覆盖了能力体系、五大城市生态、货币与银行系统、法律与治理架构、宗教与教育、核心历史（奥莱因战争）、暗界起源假说、方言系统、艺术流派、吟游诗人文学，以及最高机密等级的"玻利瓦尔真相"与"神话位阶"档案。

叙事风格

无论做视频还是写设定，我追求的核心原则是：信息密度与叙事节奏的平衡。

游戏考据视频以严密的逻辑链条和一手资料为骨架，拒绝空洞的情绪输出；世界观构建则从社会学视角切入——将经济学（维尔的两本理论著作）、政治学（元老会议的三方博弈）、法律史（法典统一战争）等维度融入奇幻设定，让虚构世界拥有"非虚构"级别的可信度。

在表达方式上，偏好冷峻、克制的语调，用事实和逻辑本身的力量驱动叙事，而非依赖情绪渲染。

创意栈：谁决定了"做什么"和"做成什么样"

在传统的网页开发中，"甲方"给出模糊的需求，"乙方"反复猜测并执行。而在我们的协作中，角色分工更加精确：

决策类型谁主导具体表现
模块方向（"这个页面该有什么"）人类比如"B站数据那里加个互动排序和雷达对比图"、"小说页加性格测试和人物关系图"
审美风格（"看起来是什么感觉"）人类比如"用中世纪告示板风格"、"去Emoji纯文字"、"调成三角布局"
内容质量把关（"这个分析到位了吗"）人类比如"彼得不是教师的孙子，是教师之子"、"加上翻译"、"这段的缺点复盘少了"
代码实现（"怎么用CSS/JS做出来"）AICSS Grid布局、SVG雷达图、手风琴交互、Python HTML转换脚本
文案生成（"把设定集/数据提炼成网页文字"）AI基于用户提供的Word文档、口述数据、up主文案，生成网页上的分析文字
Bug修复（"为什么这个按钮点了没反应"）人机协作用户发现并描述问题 → AI诊断并提出修复方案 → 用户验证反馈 → AI迭代修复
结构优化（"页面太长改成卡片折叠"）人机协作用户提出体验问题 → AI设计交互方案 → 用户确认方向 → AI执行重构

这个网站不是一次性设计出来的，而是经历了四次创意跃迁。每次跃迁都不是技术驱动的，而是我问了自己一句"这个页面还能做什么？"：

阶段一：静态展示站

初始需求

最早的构想是一个标准作品集网站：首页 + 关于 + B站数据 + 设定集陈列 + 小说页面。结构是静态的，功能是展示型的。AI 的角色是"听话的翻译器"——用户描述需求，AI 转化为 HTML/CSS。这个阶段产出了基础骨架：导航栏、Hero区、导航卡片、内容区域、页脚——这些今天仍在用的模块。

阶段二：交互化转型

转折点：用户说"那个能力者雷达图能不能让人拖动滑块然后出评估结果？"

这是整个网站气质转变的决定性时刻。不再满足于"展示已经完成的作品"，而是想让面试官在这个网站上做点什么。从此"交互"成为新增模块的默认要求。催生了属性雷达评估系统、战斗伤害计算器、掉落模拟器、公会任务交互式决策。AI 从"翻译器"升级为"设计方案的提出者"——用户说"要个伤害公式计算器"，AI 需要自己设计公式、参数、布局和输出格式。

阶段三：内容升维

转折点：用户把7期视频的完整Word文案发给 AI

这是信息密度的一次暴增。在此之前 B站页面只有标题和播放量。在此之后：每期有了100字研究摘要 → 7款游戏的完整拆解 → 文案本身的元分析（叙事结构、风格演变、缺点复盘、数据验证）。AI 升级为"内容合作创作者"——不只翻译用户的话，而是基于原始素材独立完成信息提取、结构设计、分析写作，交给用户审核修改。

阶段四：审美打磨与元反思

转折点：用户说"排版变成三角形""加一个缺点复盘""首页那块太重复了"

当前阶段的核心工作不再是"加新东西"，而是删、调整、打磨、反思。三角布局的CSS冲突调试、手风琴组件的toggle事件Bug修复、英国编剧原话的翻译补充、数据验证的交叉对比——这些都是典型的"最后20%的细节占了80%的时间"。AI 承担了一个新角色："有记忆的合作者"——因为 AI 记住了之前每一次对话的上下文，当用户说"用卡特原话替换掉那个note"时，不需要被重新告知卡特是谁。

阶段五：交互伴侣系统 · 从"作品集"到"体验" 当前

转折点：用户说"能不能设计一个我的对话小人在里面？"

这是整个网站最游戏化的一次迭代。从首页像素小人的对话框开始，逐步演变为一个完整的交互伴侣系统：可拖拽小人（含缓动跟随、边缘碰撞、松手弹跳、移动端长按防误触）、三层对话递进（欢迎→彩蛋→打破第四面墙）、空闲漫游模式、速度检测彩蛋、靠近模块介绍、拖到卡片上方"跟随进入"二级页面、跨页面访客小人（localStorage 状态传递）。AI 的角色升级为"交互设计师 + 游戏系统策划"——不再只写代码，而是设计对话记忆去重、连锁对话时序、漫游缓动曲线等行为系统。同时做了全站搜索、B站实时数据同步（JSONP）、手机汉堡菜单、回到顶部按钮等基础设施。整个系统的代码量翻了一倍以上，但用户的操作感受反而更轻了——这正是好的交互设计要达到的效果。

关键架构决策

为什么不用框架（React/Vue）？这是一个静态作品集网站，不是Web应用。引入框架会带来不必要的构建步骤和依赖管理。纯HTML/CSS/JS意味着任何电脑的浏览器都能直接打开。但代价是页面数量增长后，导航栏等公共部分需借助sed批量操作——这在框架中本应是一个组件。
为什么CSS分散在各页面？理想情况下应有一个统一样式表。但实际迭代中，每个页面的独有样式是逐步叠加的，AI每次编辑时直接内联写入更高效。更大的问题是外部style.css中的全局规则会与页面专属样式冲突——三角形布局调试就是典型案例。最终采用"style.css提供全局骨架 + 各页面提供专属皮肤"的混合架构。
引入Python BeautifulSoup的代价。BS4在"游戏拆解页面重组为折叠卡片"时，第一次转换丢失了所有正文内容——因为脚本的"remaining children"提取逻辑有Bug，7个分析卡片只剩标题。教训：对结构化HTML做批量转换时，先用一个元素验证脚本逻辑，确认无误后再跑全量。
为什么不用拖拽库（interact.js / draggable）？像素小人的拖拽系统——含缓动跟随、边缘碰撞、速度检测、移动端长按防误触——看起来适合用一个现成JS库。但实际需求中包含大量非标准交互：拖拽时切换图片、松手弹跳动画、靠近模块时触发对话链、双击回家、漫游模式……这些都不是拖拽库的常规功能。用原生事件实现虽然经历了四次重写，但换来了完全的行为可控性——每个缓动系数、每个阈值、每条对话的触发时机都可以精确调整。对于作品集这种需要"性格"的项目，这比引入一个通用库更有价值。
B站实时数据同步的五次技术方案迭代。用户提出"让主页粉丝数和播放量跟B站实时同步"。这个问题在纯静态GitHub Pages上看起来无解——B站API有跨域限制，前端fetch会被CORS拦截。以下是完整的五次尝试：

方案一 Vercel Serverless（❌）：部署api/bili-stats.js代理函数，服务器端调用B站API规避跨域。问题是Vercel默认域名在国内被墙，HR访问不了。

方案二 Cloudflare Workers（❌）：Worker脚本处理B站API代理，国内理论可访问。但Cloudflare编辑器持续加载失败，无法进入代码页面。

方案三 公共CORS代理（❌）：corsproxy.io和allorigins.win等免费服务。国内网络超时，两个代理全部不可达。

方案四 腾讯EdgeOne Pages（❌）：国内可访问的serverless平台。但EdgeOne的serverless函数只支持框架项目（Next.js/Nuxt等），纯HTML站不认node-functions目录。尝试api/、functions/、node-functions/三种目录命名，均404。

方案五 GitHub Actions（✅ 采纳）：我提出"能不能用GitHub Action自动爬取"。这个思路绕过了所有核心矛盾——Actions运行在GitHub的美国服务器上，既不受国内墙影响，也不受浏览器跨域限制。Python脚本每30分钟自动调用B站API获取真实粉丝数，解析JSON后更新index.html和bilibili.html中的数值，自动commit并push到仓库，触发GitHub Pages重新部署。整个过程对页面访客完全透明——页面始终秒开静态数字，后台静默同步。

教训：方案一至四都是同一个思路的不同实现——"找一个免费的、国内能访问的serverless平台跑代理"。这个组合在中国大陆互联网环境中不存在。方案五换了一个完全不同的角度——"不在运行时解决，在构建时解决"。这是AI反复建议"算了"但我坚持继续探索后才找到的突破。有时候人类直觉中的"应该还有办法"比AI的"技术上不可行"更值得信任。

可复用的协作方法论

原则一：人类负责"做什么"和"好不好"，AI负责"怎么做"

这是整段协作中最稳定的分界线。用户不需要告诉AI技术细节；AI不需要替用户判断审美。当AI的方案被否决时（比如三角形布局第一次没生效），AI应该换一种技术思路而非坚持原方案。

原则二：一次只做一件事，但把这件事做完

每个交互模块的添加都遵循同一节奏：需求 → 设计 → 实现 → 验收 → 修复 → 再验收。AI 不会在加掉落模拟器的路上顺便把一个视频卡片的颜色改了。这种"窄而深"的推进方式确保每次改动可追溯。

原则三：给AI看原始素材，让人做最终裁决

当用户把Word文档直接发给AI时，产出质量显著高于"你帮我写一个细胞分裂的分析"。因为AI能看到原始材料中的细节。但当AI从素材中提炼出内容后，人的审核不可跳过——彼得是"教师之子"而非"教师的孙子"这类只有人类才知道的正确信息必须由人来纠正。

原则四：失败是协作的养料，前提是记录每一次失败

三角布局的CSS冲突、BS4内容丢失、toggle事件异步Bug、全部展开按钮失效——这些不是AI的"错误"，而是协作过程中的可复用教训。把每一次修复都记录下来，下一段人机协作的效率和可靠性都会有显著提升。

原则五：交互系统需要"手感"，而不只是功能

像素小人拖拽系统经历了四次重写：从 position:fixed 坐标不准，到 transform 累积位移受滚动影响，再到最终 fixed + absolute 坐标系转换。每个版本在功能上都"能拖"，但手感完全不同。最终版加入了缓动跟随（ease 0.35）、松手弹跳动画、边界软碰撞、移动端长按阈值——这些都不是功能需求，而是体验需求。在人机协作中，AI 应该主动提出体验优化方案（"要不要加个缓动，让拖拽更有物理感？"），而不是等用户发现"手感不好"。
