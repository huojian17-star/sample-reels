关于这份网页本身

一份由人类导演、AI执行的作品集网站——从创意构想、架构设计到代码落地的完整协作记录

项目全貌

13独立页面
~1,600行 CSS
~6,000行 JavaScript
9交互式模块
1人类 + 1 AI

这个网站从第一行 HTML 到最终的 13 页结构，全程由一个人（我，邓恺恒）提供方向、内容、审美判断和迭代反馈，由一个 AI（Claude）执行编码、重构、调试和内容生成。下面记录的是这段协作是如何工作的，以及从中可以提取什么可复用的方法论。

创意栈：谁决定了"做什么"和"做成什么样"

在传统的网页开发中，"甲方"给出模糊的需求，"乙方"反复猜测并执行。而在我们的协作中，角色分工更加精确：

决策类型谁主导具体表现
模块方向（"这个页面该有什么"）人类比如"B站数据那里加个互动排序和雷达对比图"、"小说页加一个性格测试和人物关系图"
审美风格（"看起来是什么感觉"）人类比如"用中世纪告示板风格"、"去Emoji纯文字"、"调成三角布局"
内容质量把关（"这个分析到位了吗"）人类比如"彼得不是教师的孙子，是教师之子"、"加上翻译"、"这一段的缺点复盘少了一些"
代码实现（"怎么用CSS/JS做出来"）AICSS Grid布局、SVG雷达图、手风琴交互、Python HTML转换脚本
文案生成（"把设定集/数据提炼成网页文字"）AI基于用户提供的Word文档、口述数据、up主文案，生成网页上的分析文字
Bug修复（"为什么这个按钮点了没反应"）人机协作用户发现并描述问题 → AI诊断并提出修复方案 → 用户验证反馈 → AI迭代修复
结构优化（"页面太长不好，改成卡片折叠"）人机协作用户提出体验问题 → AI设计交互方案 → 用户确认方向 → AI执行重构

创意演进：从"展示型主页"到"交互式作品集"

这个网站不是一次性设计出来的，而是经历了四次创意跃迁。每次跃迁都不是技术驱动的，而是用户说了一句"能不能..."。下面按时间顺序还原这个演进过程。

阶段一：静态展示站

初始需求

最早的构想是一个标准作品集网站：首页 + 关于 + B站数据 + 设定集陈列 + 小说页面。结构是静态的，功能是展示型的，没有交互。这时候的目标是"把东西放上去"。AI 的角色是"听话的翻译器"——用户描述需求，AI 转化为 HTML/CSS。

这个阶段产出了目前网站的基础骨架：导航栏、Hero区、导航卡片、内容区域、页脚——这些今天仍然在用的模块，是在最初20分钟内搭建完成的。

阶段二：交互化转型

转折点：用户说"那个能力者雷达图能不能让人拖动滑块然后出评估结果？"

这是整个网站气质转变的决定性时刻。用户不再满足于"展示已经完成的作品"，而是想让面试官在这个网站上做点什么。从此之后，"交互"成为了新增模块的默认要求。

这个阶段催生了：属性雷达评估系统、战斗伤害计算器、掉落模拟器、公会任务交互式决策流程。AI 的角色从"翻译器"升级为"设计方案的提出者"——用户说"想要个伤害公式计算器"，AI 需要自己设计公式、参数、滑块布局和输出格式，然后交给用户验收。

阶段三：内容升维

转折点：用户把7期视频的完整Word文案发给 AI，说"把这些也做进网站里"

这是信息密度的一次暴增。在此之前，B站页面只有标题和播放量。在此之后，每期视频有了100字的研究摘要；然后有了整篇的游戏拆解（7款游戏 × 4维度分析）；再然后有了文案本身的元分析（叙事结构、风格演变、缺点复盘、数据验证）。

AI 在这个阶段的角色又一次转变——从"设计方案提出者"升级为"内容合作创作者"。AI 不只是把用户的话翻译成 HTML，而是基于用户的原始素材（Word文档），独立完成信息提取、结构设计、分析写作，然后交给用户审核修改。

阶段四：审美打磨与元反思

转折点：用户说"首页那块太重复了""排版变成三角""加一个缺点复盘""做一个关于这个网页本身的页面"

当前阶段的核心工作不再是"加新东西"，而是删、调整、打磨、反思。三角布局的CSS冲突调试、手风琴组件的toggle事件Bug修复、英国编剧原话的翻译补充、每一期文案的数据验证……这些都是典型的"最后20%的细节占了80%的时间"。

AI 在这个阶段承担了一个新的角色："有记忆的合作者"。因为AI记住了之前每一次对话的上下文，所以当用户说"用卡特原话替换掉那个note"时，AI不需要被重新告知卡特是谁、那段话是什么——它直接找到了正确的代码位置并执行了修改。

技术栈：这份网页是用什么和怎么搭建的

前端技术

技术用途为什么选它
纯 HTML/CSS/JS全部页面零依赖、零构建步骤、面试官可以直接右键查看源码。不需要npm install，不需要等待编译。
CSS Grid + Flexbox布局系统三角布局、卡片网格、响应式断点全部依赖原生CSS布局，不引入Bootstrap等重量框架。
SVG（内联）雷达图、人物关系图可编程的矢量图形，JS直接操作DOM节点生成和更新，不需要Canvas或图表库。
CSS自定义属性全局主题色--accent、--text-dim 等变量统一管理颜色，换主题只需改十几行。
Media Queries响应式适配768px断点，桌面三角布局→移动单列堆叠，纯CSS无JS。
details/summary + JS折叠卡片原生HTML折叠元素 + 手风琴JS增强。不使用任何UI库。

后端/辅助工具

工具用途为什么选它
Python (python-docx)Word文档内容提取用户的7期视频文案以.docx格式提供，需要用python-docx读取段落文本后转为网页内容。
Python (BeautifulSoup)HTML结构批量重构游戏拆解页面从"长篇滚动"重构为"折叠卡片"时，手动改7个analysis-block不现实，用BS4自动转换。

架构决策记录

决策一：为什么不用框架（React/Vue）？
这是一个静态作品集网站，不是Web应用。引入框架会带来不必要的构建步骤、依赖管理和学习成本。纯HTML/CSS/JS意味着：任何一台电脑上的浏览器都能直接打开，面试官右键就能看到完整的、未混淆的源码。这对于展示技术透明度是加分项。但代价是：当页面数量增长到13个时，导航栏等公共部分的修改需要批量sed操作——这在框架中本应是一个组件。
决策二：为什么把CSS写在每个页面的<style>里而不是全部放在style.css？
理想情况下应该有一个统一的样式表。但在这个项目的实际迭代中，每个页面的独有样式（游戏设计页的中世纪木板、文案拆解页的折叠卡片、首页的三角形布局）是逐步叠加的，AI在每次编辑时直接内联写入更高效。更大的问题是：外部style.css中的某些全局规则（如.nav-cards的auto-fit）会与页面专属样式冲突——三角形布局的调试过程就是最典型的案例。最终采用"style.css提供全局骨架 + 各页面<style>提供专属皮肤"的混合架构。
决策三：什么时候用Python脚本，什么时候手动改？
当一个操作需要重复3次以上时，写脚本。例如：13个页面的导航栏同步更新（sed一行）、7个analysis-block的结构转换（BeautifulSoup脚本）。当操作只发生一次且内容高度依赖人的判断时，手动改——例如每一期文案拆解的具体分析内容，即使有7期，也需要AI逐期阅读原始文案后独立撰写，无法模板化。
决策四：引入Python BeautifulSoup的代价。
BS4在"游戏拆解页面重组为折叠卡片"的任务中，第一次转换时丢失了所有正文内容，导致7个分析卡片只剩下标题——因为脚本的"remaining children"提取逻辑有Bug。这个Bug的修复又耗费了一轮迭代。教训是：对结构化HTML做批量转换时，先用一个元素验证脚本逻辑，确认无误后再跑全量。这个教训本身也是人机协作的一部分——用户发现了"内容没了"，AI诊断出是BS4脚本的提取逻辑问题，然后重建了页面。

可复用的协作方法论

原则一：人类负责"做什么"和"好不好"，AI负责"怎么做"

这是整段协作中最稳定的一条分界线。用户不需要告诉AI"用grid-column: 2来实现居中"；AI不需要替用户判断"这个设计是不是太啰嗦了"。

最佳实践：用户用自然语言描述需求和审美判断（"这里太重复了，去掉"、"改成三角形"），AI提出技术方案并执行。当AI的方案被用户否决时（比如三角形布局第一次没生效），AI应该换一种技术思路而非坚持原方案。

原则二：一次只做一件事，但把这件事做完

每个交互模块的添加都遵循同一节奏：需求 → 设计 → 实现 → 验收 → 修复 → 再验收。用户说"加一个掉落模拟器"——AI 不会在完成的路上又顺手把一个视频卡片的颜色改了。这种"窄而深"的推进方式确保了每次改动都是可追溯的，出了问题也知道是哪个步骤引入的。

原则三：人机之间最有效的沟通是"给AI看原始素材，让人做最终裁决"

当用户把Word文档直接发给AI时，产出质量显著高于"你帮我写一个细胞分裂的分析"。因为AI能看到原始材料中的细节（克林顿·霍金的故事、卡特回应的精确措辞），而不是从训练数据中猜测。但当AI从原始素材中提炼出内容后，人的审核是不可跳过的步骤——彼得是"教师之子"而非"教师的孙子"、"那封邮件截图命名为编剧访谈"——这些只有人类才知道的正确信息，必须由人来纠正。

原则四：失败是协作的养料，前提是记录每一次失败

三角布局的CSS冲突、BS4内容丢失、toggle事件异步Bug、全部展开按钮失效——这些失败不是AI的"错误"，而是协作过程中的可复用教训。如果把每一次修复都记录下来（就像这个页面做的那样），下一段人机协作的效率和可靠性都会有显著提升。
